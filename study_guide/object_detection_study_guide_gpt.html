PART 01

全局定位：检测任务到底多了什么？

对应 WI2022 Lecture 13：Slides 31–38；Lecture 14：Slides 8–11

图像分类只需要回答一个问题：这张图是什么类别？ Object Detection 则要回答三个问题：有什么类别、每个物体在哪里、每类有多少个实例。所以它天然比分类更麻烦：输出数量不是固定的，每个输出还同时包含离散类别与连续坐标。

检测任务处于分类和实例分割之间：它不要求像素级 mask，但要求每个实例的类别和边界框。
核心定义 Object Detection 的输入是单张 RGB 图像；输出是一组 detected objects。对每个对象，需要预测：类别标签和 bounding box。课程 slides 明确强调检测会遇到三个挑战：输出数量可变、输出类型混合、图像分辨率通常比分类更高。

变量数量

一张图可能有 0 个、1 个或几十个物体，不能像分类那样固定输出一个 1000 维向量。

混合输出

类别是离散变量，框坐标是连续变量，所以 loss 通常由 classification loss + regression loss 组成。

高分辨率

检测小物体需要保留空间细节，常用更大的输入分辨率；这会直接带来显存和速度问题。

Modal vs Amodal 常规检测里 bounding box 通常包住可见部分，这叫 modal box。Amodal box 则试图包住物体完整范围，即使一部分被遮挡。大多数主流检测数据集和作业默认用 modal box。别小看这个差别：遮挡场景下，标注协议不同会直接影响模型学到的“物体范围”。
PART 02

Bounding Box 与 IoU：检测的几何语言

对应 WI2022 Lecture 13：Slides 34–45

检测器最基本的几何对象是 axis-aligned bounding box。常见表示有两种：(x, y, w, h) 或 (x1, y1, x2, y2)。作业和框架之间最容易出 bug 的地方，就是坐标格式没对齐。

IoU 也叫 Jaccard similarity。它是训练正负样本、NMS、mAP 评价的共同底层语言。
IoU formula
IoU(A, B) = Area(A ∩ B) / Area(A ∪ B)
IoU 数值直觉解释课程中的经验说法
≈ 0.5预测框和 GT 框大致对上了，但边界还比较粗decent
≈ 0.7框已经比较准确pretty good
≈ 0.9几乎贴合 GT 框almost perfect
def box_iou_xyxy(boxes1, boxes2):
    """
    boxes1: (N, 4), boxes2: (M, 4), format = (x1, y1, x2, y2)
    return: (N, M) IoU matrix
    """
    lt = torch.maximum(boxes1[:, None, :2], boxes2[None, :, :2])  # left-top
    rb = torch.minimum(boxes1[:, None, 2:], boxes2[None, :, 2:])  # right-bottom

    wh = (rb - lt).clamp(min=0)
    inter = wh[:, :, 0] * wh[:, :, 1]

    area1 = (boxes1[:, 2] - boxes1[:, 0]).clamp(min=0) * \
            (boxes1[:, 3] - boxes1[:, 1]).clamp(min=0)
    area2 = (boxes2[:, 2] - boxes2[:, 0]).clamp(min=0) * \
            (boxes2[:, 3] - boxes2[:, 1]).clamp(min=0)

    union = area1[:, None] + area2[None, :] - inter
    return inter / union.clamp(min=1e-6)
常见 bug

把 (x, y, w, h) 当成 (x1, y1, x2, y2) 算 IoU，结果出现负面积或 IoU 全是 0。

调试方式

打印前 5 个 box，手算一个样例；确认 x2 > x1、y2 > y1，并画到图上。

PART 03

NMS 与 mAP：从一堆框到可评价结果

对应 WI2022 Lecture 13：Slides 77–96；Lecture 14：Slides 98–110

检测器原始输出常常不是“干净的一组对象”，而是一大堆高度重叠的框。比如一只狗周围可能被预测出十几个 dog 框，每个分数都不低。Non-Max Suppression（NMS）就是把这些冗余框压成少数代表框的后处理。

NMS 是“贪心去重”：先保留最高置信度框，再删除与它重叠过高的框。
NMS 的缺陷 如果两个真实物体本来就高度重叠，NMS 可能把其中一个“好框”误删。人群、遮挡、密集小目标场景里这会很痛。现代方法会用 Soft-NMS、学习式 NMS，或者用 DETR 一类集合预测思路减少手工后处理，但经典检测器里 NMS 仍是基本功。

mAP：检测器怎么评分？

mAP 的核心逻辑是：先对每个类别单独算 AP，再对所有类别取平均。AP 是 Precision-Recall 曲线下面积。课程里给出的流程是：对所有测试图运行 detector 和 NMS；对每个类别，把该类别所有 detections 按分数从高到低排序；每个 detection 若能以足够 IoU 匹配某个未匹配 GT，就记为 TP，否则记为 FP；逐步画出 PR 曲线并计算面积。

AP 是一个类别的面积；mAP 是多个类别 AP 的平均值。PASCAL VOC 常用 IoU > 0.5，COCO 会用多个 IoU 阈值更严格地平均。
PART 04

R-CNN：把分类器强行改造成检测器

对应 WI2022 Lecture 13：Slides 61–76；Lecture 14：Slides 9–11

R-CNN 的思想很直接：既然我们已经有强大的图像分类 CNN，那么能不能先找出一堆可能含有物体的区域，然后把每个区域裁成固定大小，逐个送进 CNN 分类？答案是能，而且在早期效果很好；但它的代价是非常慢。

Input image→ Region proposals
≈ 2k RoIs→ Warp each region
224×224→ CNN forward
每个 RoI 一次→ Class + Bbox
R-CNN 的核心瓶颈：一张图大约 2000 个候选区域，就要做大约 2000 次 CNN forward。

R-CNN 为什么它是重要的第一步？

它把检测问题拆成两个子问题：哪里可能有物体由 proposal method 负责；这个区域是什么物体由 CNN 分类器负责。这样就把“可变数量输出”的难题转化成“对很多候选区域做分类”。

R-CNN 的训练标签

对每个 region proposal，使用它与 GT box 的 IoU 判定训练类型：

类型判定规则训练目标
Positive与某个 GT box 的 IoU > 0.5预测该 GT 的类别，并学习 bbox transform
Negative与所有 GT box 的 IoU < 0.3预测 background，不训练 box transform
NeutralIoU 在 0.3 到 0.5 之间通常忽略，避免模糊样本干扰训练

Bounding Box Regression：不是直接重画框，而是预测修正量

给定 proposal box p 和 GT box g，模型通常预测一个 transform (tx, ty, tw, th)，把 proposal 移动、缩放成更接近 GT 的框。

tx = (gx - px) / pw,   ty = (gy - py) / ph,   tw = log(gw / pw),   th = log(gh / ph)
R-CNN 的根本问题 候选框高度重叠，同一批像素会被 CNN 反复处理。R-CNN 慢，不是因为分类头太重，而是因为 backbone 的卷积计算被重复了几千次。
PART 05

Fast R-CNN：共享卷积特征，别重复算

对应 WI2022 Lecture 14：Slides 12–22，49–50

Fast R-CNN 的想法非常漂亮：不要对每个 proposal 单独跑 CNN，而是整张图只跑一次 backbone，得到 feature map；然后把 proposal 投影到 feature map 上，裁剪出对应区域特征，再用轻量 per-region network 做分类和框回归。

Fast R-CNN 的关键是计算复用：backbone 对整图只算一次，region-level head 相对轻量。
架构部分作用典型例子
Backbone从整张图抽取共享 feature mapAlexNet / VGG / ResNet
RoI feature extractor把每个 proposal 的区域特征裁成固定大小RoI Pool / RoI Align
Per-region network对每个区域预测类别和 bbox offsetFC layers 或 ResNet 最后一阶段
这一步的设计哲学 检测不是“把分类模型复制 2000 次”，而是共享密集特征，再对少量候选区域做轻量决策。这个思想后来贯穿 Faster R-CNN、Mask R-CNN、FPN 乃至很多现代视觉模型。
PART 06

RoI Pool / RoI Align：如何裁剪特征图？

对应 WI2022 Lecture 14：Slides 23–48

Fast R-CNN 有一个关键问题：proposal 是在原图坐标里给出的，但我们要在 feature map 上裁剪特征。由于 CNN 存在 stride / pooling，原图与 feature map 的空间分辨率不同，但它们之间仍然有坐标对应关系。

RoI feature extraction：把原图 proposal 投影到 feature map，再裁成固定空间尺寸。

RoI Pool

把投影后的 RoI “吸附”到 feature grid 上，再划分为固定数量 bins，每个 bin 做 max pooling。

问题：量化坐标会造成位置误差，尤其对小目标和分割任务更明显。

RoI Align

不做 snapping；在浮点坐标上采样，用 bilinear interpolation 得到特征，再池化。

优点：坐标对齐更精确，是 Mask R-CNN 等方法的重要细节。

f(x, y) = Σi,j f(i,j) · max(0, 1 − |x − i|) · max(0, 1 − |y − j|)
为什么这块对代码很重要？ RoI Align 通常要求 boxes 的坐标、feature map 的 stride、batch index 都正确。检测作业里很多“不报错但训练不动”的 bug，其实是 box 坐标投影错了：模型看见的区域根本不是物体。
PART 07

Faster R-CNN：让 CNN 自己产生候选框

对应 WI2022 Lecture 14：Slides 51–72，84

Fast R-CNN 虽然避免了重复跑 backbone，但它仍然依赖外部 proposal method。这些 proposal 往往跑在 CPU 上，成为新的瓶颈。Faster R-CNN 的关键改进是：插入 Region Proposal Network（RPN），直接从 CNN feature map 预测候选框。

Image→ Backbone feature map→ RPN
objectness + box transform→ Proposals→ RoI Align→ Second-stage
class + bbox
Faster R-CNN 仍是 two-stage detector：第一阶段生成 proposals，第二阶段精分类和精回归。

Anchor：把“哪里可能有物体”转成 dense prediction

RPN 会在 feature map 每个位置放置若干个不同尺度、长宽比的 anchor。对每个 anchor，RPN 预测两件事：它是否包含物体，以及如何修正它变成更好的 proposal。

RPN 把候选框生成变成卷积层上的密集预测：每个位置、每个 anchor 都有 objectness 和 box offset。

Faster R-CNN Two-stage 的优势

第一阶段粗略提出少量高质量候选框，第二阶段对这些候选框做更精细分类与回归。因此它通常精度强，尤其对复杂场景更稳；代价是 pipeline 更复杂、速度通常慢于单阶段检测器。

PART 08

Feature Pyramid Network：多尺度怎么处理？

对应 WI2022 Lecture 14：Slides 73–83

检测需要识别各种尺度的物体：远处的小人、近处的大车。经典办法是 image pyramid：把同一张图 resize 成多个尺度，每个尺度单独跑 detector。但这非常昂贵，因为不同尺度之间几乎不共享计算。

FPN 用高层语义自顶向下补回低层特征，得到高分辨率且高语义的多尺度特征。
为什么 FPN 如此常用？ 它把“多尺度图像金字塔”的思想搬到特征空间里：只跑一次 backbone，却能在不同分辨率的 feature levels 上检测不同大小的物体。Faster R-CNN、RetinaNet、FCOS 都可以和 FPN 结合。
PART 09

Single-Stage Detectors：一步到位的检测器

对应 WI2022 Lecture 14：Slides 84–97

Faster R-CNN 的第二阶段要对每个 proposal 裁特征、分类、回归。于是自然会问：我们真的需要第二阶段吗？ Single-stage detector 直接在密集特征图上预测类别和 box，不再生成候选框后再精分类。

Two-stage

代表：Faster R-CNN

Stage 1 产生 proposals，Stage 2 精分类和精回归。

优点：通常更稳、更准确；缺点：结构和推理流程更复杂。

Single-stage

代表：YOLO / SSD / RetinaNet / FCOS

直接在 feature map 上密集预测类别和 box。

优点：速度快、结构更直接；缺点：前景/背景类别极不平衡。

RetinaNet：像 RPN，但直接预测类别

RetinaNet 的输出形式和 RPN 很像：每个 feature point 有 K 个 anchors，每个 anchor 直接预测 C + 1 类分类分数和 4 个 box transform。它的关键问题是：背景 anchors 太多，前景 anchors 太少，class imbalance 非常严重。RetinaNet 用 Focal Loss 降低容易负样本对训练的主导。

Focal Loss intuition
CE(pt) = −log(pt)   →   FL(pt) = −(1 − pt)γ log(pt)
Focal Loss 的直觉 如果一个背景框很容易被模型判断为背景，那么它的 p_t 很高，(1-p_t)^γ 很小，loss 权重就被压低。这样模型能把注意力更多放在困难样本和前景样本上。

FCOS：Anchor-free 的思路

FCOS 进一步抛弃 anchor：如果某个 feature point 落在 GT box 内，它就是 positive；模型预测该点到 box 左、上、右、下四条边的距离 L, T, R, B，再预测 centerness 抑制靠近边缘的不可靠预测。

FCOS 把 anchor 设计问题变成 point-based prediction：每个点直接预测到 box 边界的距离。
模型是否 two-stage是否 anchor-based核心问题 / 设计
Faster R-CNN是是RPN 生成 proposals，第二阶段精修
RetinaNet否是密集 anchors + Focal Loss 处理类别不平衡
FCOS否否点分类 + 边界距离回归 + centerness
PART 10

代码架构：张量形状、Loss 与 Debug

面向 A4 / HW Object Detection 的工程视角

检测代码最难的不是“写一个 CNN”，而是管理大量张量：anchors、proposals、GT boxes、class logits、box offsets、batch index、NMS 结果。你必须用 shape 统治它们。

Faster R-CNN 的典型模块拆分

class FasterRCNN(nn.Module):
    def __init__(self, backbone, rpn, roi_pool, roi_head):
        super().__init__()
        self.backbone = backbone      # image -> feature map
        self.rpn = rpn                # feature map -> proposals
        self.roi_pool = roi_pool      # proposals -> fixed-size features
        self.roi_head = roi_head      # region features -> class logits + bbox deltas

    def forward(self, images, targets=None):
        feats = self.backbone(images)             # (N, C, H, W)
        proposals, rpn_losses = self.rpn(feats, targets)
        roi_feats = self.roi_pool(feats, proposals)  # (num_rois, C, 7, 7)
        detections, roi_losses = self.roi_head(roi_feats, proposals, targets)
        return detections, {**rpn_losses, **roi_losses}

几个必须盯住的 shape

张量典型形状含义常见错误
feature map(N, C, H, W)backbone 共享特征忘记 stride，导致 anchors 映射错位
anchors(N, H×W×K, 4)每个位置 K 个预设框K、H、W flatten 顺序不一致
RPN logits(N, 2K, H, W)每个 anchor 是否是 objectreshape 时类别维和 anchor 维混在一起
RPN deltas(N, 4K, H, W)每个 anchor 的坐标修正dx/dy/dw/dh 顺序与解码函数不一致
RoI features(R, C, 7, 7)R 个 proposal 的固定大小特征batch index 缺失，多个 batch 的 proposal 串了
cls logits(R, C+1)区域类别，含 background忘记 background 类，label 越界
bbox deltas(R, 4C) 或 (R, 4)类别相关或类别无关 bbox 修正训练和推理使用的格式不一致

Loss 通常分成四块

🎯
RPN cls

anchor objectness：前景 / 背景

📦
RPN reg

positive anchors 的 box transform

🏷️
RoI cls

🔧
RoI reg

COMPETITIVE EDGE

Vibe Coding 时代的竞争力

Object Detection 是特别适合训练“真能力”的章节，因为它不像分类网络那样一眼就能调库跑通。检测器里有大量几何、张量、采样、后处理和评价协议；AI 可以帮你写代码，但很容易在这些细节里悄悄写错。

场景：AI 写了 IoU，但训练正样本几乎为 0 你需要立刻怀疑：坐标格式是否混了？box 面积是否有负数？IoU 阈值是否太高？anchors 是否没有覆盖 GT？如果你知道 positive / negative assignment 的机制，就不会只盯着 loss 发呆。
场景：模型输出很多重复框 这不一定是模型坏了，可能只是 NMS 没做好：阈值过高、按类别 NMS 没分开、score 排序错、坐标没 decode。能把“模型预测”和“后处理错误”分开，是工程调试能力。
场景：面试让你解释 Faster R-CNN 浅层答案是“RPN + Fast R-CNN”。深层答案要能讲清楚：共享 backbone、anchor dense prediction、RPN objectness 和 regression、RoI Align、two-stage 的 trade-off、FPN 如何处理尺度。再进一步，能说出每个 head 的输出 shape，就很稳了。
学习建议：这章不要只背模型名字。你真正要掌握的是一套检测器设计范式：
dense candidates → assign labels by IoU → predict class + box → decode boxes → NMS → mAP evaluation。这套范式学透后，YOLO、SSD、RetinaNet、FCOS、DETR 都更容易理解。
