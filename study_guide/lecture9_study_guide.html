PART 01

CPU vs GPU：为什么深度学习需要 GPU？

对应 Slides 3–10

Justin 在 Slide 10 用了一张非常直观的对比表格。核心区别不在于哪个"更好"，而在于它们的设计哲学完全不同：

CPU (Ryzen 9 3950X)GPU (NVIDIA Titan RTX)
核心数16 (32 threads)4608
时钟频率3.5 GHz (boost 4.7)1.35 GHz (boost 1.77)
内存System RAM（可扩展到数百 GB）24 GB GDDR6（固定）
价格$749$2499
FP32 算力~4.8 TFLOP/s~16.3 TFLOP/s
核心直觉 CPU 像一小队精英特种兵——每个核心都很强、很灵活，擅长复杂的顺序任务（分支预测、缓存优化、乱序执行）。

GPU 像一支万人大军——每个核心很"笨"（没有分支预测，没有复杂缓存），但有数千个同时工作，擅长大规模并行任务。

深度学习的核心操作——矩阵乘法——恰好是最典型的并行任务：输出矩阵的每个元素可以独立计算。这就是为什么 GPU 在深度学习中如此重要。

为什么矩阵乘法天然适合 GPU？

假设你要计算 C = A × B，其中 A 是 1000×1000，B 是 1000×1000。输出 C 有 100 万个元素，每个元素是一个 1000 维的点积。这 100 万个点积之间完全独立——它们不需要互相通信，不需要等待前一个完成。GPU 的 4608 个核心可以同时计算上千个点积，而 CPU 的 16 个核心只能同时处理 16 个。

在深度学习中，几乎所有核心操作都是矩阵运算：全连接层是矩阵乘法，卷积可以通过 im2col 转化为矩阵乘法，BatchNorm 是逐元素操作（也是并行的）。所以 GPU 在深度学习中通常比 CPU 快 10–100 倍。

PART 02

GPU 内部结构与 Tensor Core

对应 Slides 11–20

Justin 在 Slide 20 展示了 RTX Titan 的内部结构。理解这些细节虽然不直接影响你写 PyTorch 代码，但能帮你理解为什么某些操作快、某些慢。

GPU 的层级结构

以 RTX Titan 为例，它包含 72 个 Streaming Multiprocessors (SM)。每个 SM 内部有 64 个 FP32 CUDA 核心和 8 个 Tensor Core。整体算力的计算：

FP32 算力：72 SM × 64 cores × 2 FLOP/cycle × 1.77 GHz = 16.3 TFLOP/s

Tensor Core 算力：72 SM × 8 Tensor Cores × 128 FLOP/cycle × 1.77 GHz = 130 TFLOP/s（！）

Tensor Core 是什么？ Tensor Core 是专门为矩阵乘加运算（Matrix Multiply-Accumulate）设计的硬件单元。它能在一个时钟周期内完成一个 4×4 矩阵的 D = A×B + C 运算（128 次浮点运算）。这比用 FP32 核心做同样的计算快约 8 倍。

Tensor Core 通常使用混合精度：用 FP16（半精度）做乘法，用 FP32 做累加，既快又不损失太多精度。这就是为什么现代训练框架默认开启 Mixed Precision Training（AMP）。
现代视角：2025 年的 GPU 这份 slides 是 2019 年的数据。到 2025 年，NVIDIA H100 的 Tensor Core 算力已达 ~990 TFLOP/s (FP16)，是 RTX Titan 的约 7.6 倍。但核心的设计理念不变：大量并行核心 + 专用矩阵运算单元。你在这门课中学到的关于 GPU 并行性的直觉，在 2025 年依然完全适用。

GPU 编程的瓶颈：内存带宽

Justin 虽然没有用大量篇幅讲这个，但这是理解 GPU 性能的关键。GPU 的计算速度往往不是瓶颈，数据传输才是。数据需要经历两个层级的传输：

CPU RAM ↔ GPU RAM（PCIe 总线）：带宽约 16 GB/s。这就是为什么你需要把数据"移动到 GPU"上（tensor.cuda()），并尽量减少 CPU-GPU 之间的数据往返。

GPU RAM ↔ GPU 计算核心（内存带宽）：RTX Titan 约 672 GB/s。即使如此，如果你的操作是"内存密集型"（读写大量数据但计算很少，比如逐元素加法），GPU 也可能被内存带宽卡住。

PyTorch 三层抽象：Tensor → Autograd → Module

对应 Slide 40

Slide 40 是整个软件部分的路线图——Justin 把 PyTorch 分成三个递进的抽象层次，并标注了它们对应的 Assignment：

Level 1: Tensor — 类似 NumPy array，但可以在 GPU 上运行 A1, A2, A3 ↓ 在此基础上加入自动微分 Level 2: Autograd — 自动构建计算图，自动计算梯度 A4, A5, A6 ↓ 在此基础上加入模块化封装 Level 3: nn.Module — 神经网络层的抽象，可存储权重和状态 A4, A5, A6
Slide 40: PyTorch 的三层递进抽象
为什么这个分层设计如此重要？ 这不仅是 PyTorch 的设计，更是你理解任何深度学习框架的思维模型。无论是 TensorFlow、JAX 还是未来的新框架，它们都遵循同样的三层结构：张量运算 → 自动微分 → 模型抽象。在你的 A1–A3 中，你手写了 forward 和 backward——那就是在 Level 1 上工作。A4 之后，你开始使用 Autograd 和 Module，让框架帮你做你之前手写的事情。
PART 05

Tensor：从 NumPy 到 GPU

对应 Slides 42–50

PyTorch 的 Tensor 和 NumPy 的 ndarray 几乎同构——API 非常相似，但有一个关键区别：Tensor 可以在 GPU 上运行。

Slide 45 的完整代码解读

Justin 在 Slide 45 展示了一个"原始"的两层网络训练循环——不用 autograd，手动计算梯度。这段代码虽然你在 A4+ 之后不会再这样写，但它是理解后续抽象层的基础：

import torch
device = torch.device('cuda')  # 指定在 GPU 上运算

N, D_in, H, D_out = 64, 1000, 100, 10
x = torch.randn(N, D_in, device=device)   # 直接在 GPU 上创建数据
y = torch.randn(N, D_out, device=device)
w1 = torch.randn(D_in, H, device=device)
w2 = torch.randn(H, D_out, device=device)

learning_rate = 1e-6
for t in range(500):
    # === 前向传播 ===
    h = x.mm(w1)              # (N, D_in) × (D_in, H) → (N, H)
    h_relu = h.clamp(min=0)   # ReLU
    y_pred = h_relu.mm(w2)    # (N, H) × (H, D_out) → (N, D_out)
    loss = (y_pred - y).pow(2).sum()

    # === 手动反向传播 ===
    grad_y_pred = 2.0 * (y_pred - y)
    grad_w2 = h_relu.t().mm(grad_y_pred)
    grad_h_relu = grad_y_pred.mm(w2.t())
    grad_h = grad_h_relu.clone()
    grad_h[h < 0] = 0                   # ReLU 的反向传播
    grad_w1 = x.t().mm(grad_h)

    # === 梯度下降更新 ===
    w1 -= learning_rate * grad_w1
    w2 -= learning_rate * grad_w2
这里的手动 backward 你应该很熟悉了 经过 A2（手写两层网络 backward）和 A3（模块化 backward），你已经能看懂这段代码的每一行。grad_h[h < 0] = 0 就是 ReLU 的反向传播——正区间梯度通过，负区间梯度归零。这正是你在 A3 中 relu_backward 里写的逻辑。

但手写 backward 在复杂网络中是不可行的——一个 ResNet-152 有几百个运算节点，手写梯度会出错且无法维护。这就引出了 Autograd。
PART 06

Autograd：自动微分的魔法

对应 Slides 55–65

Autograd 是 PyTorch 的核心引擎。Slide 55 展示了同样的两层网络，但不再手写 backward——只需要调用 loss.backward()，PyTorch 会自动帮你算出所有梯度。

import torch

N, D_in, H, D_out = 64, 1000, 100, 10
x = torch.randn(N, D_in)
y = torch.randn(N, D_out)
w1 = torch.randn(D_in, H, requires_grad=True)  # 告诉 PyTorch 追踪这个张量
w2 = torch.randn(H, D_out, requires_grad=True)

learning_rate = 1e-6
for t in range(500):
    # 前向传播（与之前完全一样！）
    y_pred = x.mm(w1).clamp(min=0).mm(w2)
    loss = (y_pred - y).pow(2).sum()

    loss.backward()  # 一行代码搞定所有反向传播！

    with torch.no_grad():         # 更新权重时不要追踪计算图
        w1 -= learning_rate * w1.grad
        w2 -= learning_rate * w2.grad
        w1.grad.zero_()           # 清空梯度（否则会累积！）
        w2.grad.zero_()

Autograd 的工作原理

当你对一个 requires_grad=True 的张量执行运算时，PyTorch 会在后台动态构建一个计算图。图中的每个节点记录了：(1) 执行了什么操作，(2) 输入是什么。当你调用 loss.backward() 时，PyTorch 从 loss 出发，沿着这个图反向遍历，用链式法则自动计算每个 requires_grad=True 张量的梯度，存储在 .grad 属性中。

三个关键机制（极易出 bug！）

① requires_grad=True：只有标记了这个属性的张量才会被追踪。输入数据 x 和标签 y 通常不需要梯度，只有权重需要。

② torch.no_grad()：更新权重时必须用这个上下文管理器包裹。否则 w1 -= lr * w1.grad 也会被当作计算图的一部分——你在"更新权重"的过程中又构建了新的计算图，导致内存泄漏和错误的梯度。

③ .grad.zero_()：PyTorch 默认累积梯度（而非覆盖）。如果你不清零，第二次 backward() 的梯度会加到第一次的梯度上。这个设计是为了支持某些高级用法（如 gradient accumulation），但对初学者来说是一个非常常见的 bug 源。

自定义 Autograd Function（Slide 65）

如果你需要定义 PyTorch 不原生支持的操作（比如一个特殊的激活函数），你可以继承 torch.autograd.Function，手动定义 forward 和 backward：

class Sigmoid(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x):
        y = 1.0 / (1.0 + (-x).exp())
        ctx.save_for_backward(y)  # 保存 y 用于 backward
        return y

    @staticmethod
    def backward(ctx, grad_y):
        y, = ctx.saved_tensors
        grad_x = grad_y * y * (1.0 - y)  # sigmoid 的导数: y(1-y)
        return grad_x
与 Assignment 3 的对应关系 你在 A3 中手写的每个 forward / backward 函数对，其实就是在做和 torch.autograd.Function 完全一样的事情。A3 教你理解原理；从 A4 开始，你使用 PyTorch 的 autograd 来自动完成这些工作。但理解原理的人在调试复杂模型时，比只会调 API 的人有巨大优势——这就是你手写 backward 的回报。
PART 07

nn.Module：模型搭建的标准范式

对应 Slides 70–85

在 Level 2（Autograd）中，权重还是"裸"张量（torch.randn(..., requires_grad=True)）。当模型变复杂时，管理几十个、几百个这样的张量会非常混乱。nn.Module 提供了一种面向对象的封装：

基本用法（Slide 75）

import torch

class TwoLayerNet(torch.nn.Module):
    def __init__(self, D_in, H, D_out):
        super(TwoLayerNet, self).__init__()
        self.linear1 = torch.nn.Linear(D_in, H)    # 权重自动创建、管理
        self.linear2 = torch.nn.Linear(H, D_out)

    def forward(self, x):
        h_relu = self.linear1(x).clamp(min=0)       # 只定义前向传播！
        y_pred = self.linear2(h_relu)
        return y_pred

# 使用
model = TwoLayerNet(D_in, H, D_out)
optimizer = torch.optim.SGD(model.parameters(), lr=1e-4)

for t in range(500):
    y_pred = model(x)
    loss = torch.nn.functional.mse_loss(y_pred, y)

    loss.backward()
    optimizer.step()       # 更新所有参数
    optimizer.zero_grad()  # 清零所有梯度

注意对比 Level 2 的代码——你不再需要手动管理 w1、w2，不再需要 torch.no_grad() 包裹更新步骤（optimizer 内部处理了），不再需要对每个张量单独调用 .grad.zero_()。

nn.Module 的三大核心机制

① model.parameters()：自动收集模型中所有的可学习参数。当你把 nn.Linear、nn.Conv2d 等层作为 Module 的属性时，PyTorch 会自动"注册"它们的权重。

② 只定义 forward()：你只需要写前向传播的逻辑。反向传播由 autograd 自动处理。永远不要手动定义 backward()。

③ Module 可以嵌套 Module：你可以在一个 Module 的 __init__ 中使用其他 Module 作为子组件。这是构建复杂网络（如 ResNet）的关键——每个 ResBlock 是一个 Module，整个 ResNet 是包含多个 ResBlock 的 Module。

自定义 Module：ParallelBlock（Slide 80）

Slide 80 展示了一个非顺序的自定义模块——两条并行的 Linear 路径，输出相乘后过 ReLU。这说明 nn.Module 不限于顺序结构，你可以用普通的 Python 代码实现任意的计算图拓扑：

class ParallelBlock(torch.nn.Module):
    def __init__(self, D_in, D_out):
        super(ParallelBlock, self).__init__()
        self.linear1 = torch.nn.Linear(D_in, D_out)
        self.linear2 = torch.nn.Linear(D_in, D_out)

    def forward(self, x):
        h1 = self.linear1(x)
        h2 = self.linear2(x)
        return (h1 * h2).clamp(min=0)  # 两路并行，结果相乘，再 ReLU

然后可以用 nn.Sequential 把多个自定义模块串联起来：

model = torch.nn.Sequential(
    ParallelBlock(D_in, H),
    ParallelBlock(H, H),
    torch.nn.Linear(H, D_out)
)
这就是 ResNet 的搭建方式 回忆 Lecture 8 的 ResNet——每个 Residual Block 就是一个自定义 Module（内部有 conv + BN + shortcut + add），然后用 Sequential 或 for 循环把多个 Block 串成完整网络。你在 A4+ 中会实际体验这种搭建方式。
PART 08

完整训练循环：optim + DataLoader

对应 Slides 75–85

把所有组件拼在一起，一个完整的 PyTorch 训练循环包含五个核心步骤。这是你从 A4 开始到未来所有项目中都会反复使用的模式：

# ============ 完整的 PyTorch 训练模板 ============
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader

# 1. 准备数据
train_loader = DataLoader(train_dataset, batch_size=64, shuffle=True)

# 2. 定义模型
model = MyModel().cuda()

# 3. 定义优化器和损失函数
optimizer = optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

# 4. 训练循环
for epoch in range(num_epochs):
    model.train()                          # 启用 BN 和 Dropout
    for images, labels in train_loader:
        images, labels = images.cuda(), labels.cuda()

        logits = model(images)             # ① 前向传播
        loss = criterion(logits, labels)   # ② 计算损失

        optimizer.zero_grad()              # ③ 清零梯度
        loss.backward()                    # ④ 反向传播
        optimizer.step()                   # ⑤ 更新参数

    # 5. 验证
    model.eval()                           # 关闭 BN 和 Dropout
    with torch.no_grad():
        for images, labels in val_loader:
            logits = model(images.cuda())
            # ... 计算验证指标
五个最常犯的错误

① 忘记 optimizer.zero_grad() → 梯度累积，loss 不正常。

② 忘记 model.train() / model.eval() → BN 用错统计量，Dropout 在测试时仍激活。

③ 忘记 torch.no_grad() → 验证时仍构建计算图，浪费内存。

④ 忘记 .cuda() → 数据在 CPU、模型在 GPU，会报 device mismatch 错误。

⑤ zero_grad() 的位置 → 放在 backward() 之前或之后都可以，但要保持一致。推荐放在 backward() 之前（上面的写法）。

PART 09

动态图 vs 静态图

对应 Slides 90–110

这是本讲的理论亮点。PyTorch 使用动态计算图（Dynamic Computation Graph），TensorFlow 1.x 使用静态计算图（Static Computation Graph）。两种范式有根本性的区别：

动态图（PyTorch）

每次前向传播都构建一个全新的计算图。backward() 执行完后图被销毁。

可以在 forward 中使用任意 Python 控制流（if/else, for 循环, while）。每次运行的图可以不同。

调试极其方便——直接用 print()、pdb、断点。

适合研究和原型开发。

静态图（TF 1.x / torch.jit）

先定义整个计算图，然后反复执行。图结构固定，不能中途修改。

控制流需要用特殊的图操作（tf.cond, tf.while_loop）来表示，不能用普通 Python if/else。

调试困难——图在构建和执行之间有一道"墙"。

但可以做图优化（算子融合、内存优化），适合部署。

动态图的核心优势（Slide 95）

Slide 95 用一个简洁的例子说明了动态图的威力：

for t in range(500):
    w2 = w2a if prev_loss < 5.0 else w2b   # 普通的 Python if!
    y_pred = x.mm(w1).clamp(min=0).mm(w2)
    loss = (y_pred - y).pow(2).sum()
    loss.backward()

在动态图中，if prev_loss < 5.0 是普通的 Python 代码。每次迭代，PyTorch 根据条件选择不同的 w2，构建不同的计算图，然后 backward 正确地计算对应图的梯度。在静态图中，这种"运行时决定网络结构"的能力需要用特殊的图操作来实现，代码会复杂得多。

Slide 110: 动态图的杀手级应用 Justin 展示了几个网络结构取决于输入的场景，这在静态图中极其痛苦，但在动态图中自然而然：

Recurrent Networks（RNN/LSTM）：序列长度因输入而异，计算图的展开深度每次都不同。
Recursive Networks：树结构网络，图的拓扑取决于输入的语法树。
Modular Networks：根据问题动态组合不同的子网络。

这些正是 PyTorch 在学术界占据主导地位的原因——研究者需要快速实验各种非标准的网络结构，动态图让这变得和写普通 Python 一样简单。

PyTorch 的折中方案：torch.jit（Slide 100）

PyTorch 也提供了 torch.jit.script 来获得静态图的好处。它会分析你的 Python 函数源码，编译成一个优化后的图对象。这在部署时很有用（模型可以脱离 Python 运行），但在研究阶段通常不需要。

PyTorch vs TensorFlow & 现代视角

对应 Slides 115–134

Slides 115–120 展示了 TensorFlow 2.0 的写法（使用 @tf.function 和 GradientTape），它已经比 TF 1.x 更接近 PyTorch 的风格了。Slide 130 介绍了 TensorBoard——一个可以可视化 loss 曲线和权重分布的工具，PyTorch 也可以通过 torch.utils.tensorboard 使用。

2025 年的现实 这份 slides 是 2019 年的，深度学习生态已经发生了很大变化：

PyTorch 已成为学术界和大部分工业界的默认选择。几乎所有顶级论文都使用 PyTorch。

TensorFlow 仍然在部署端有优势（TF Serving、TF Lite、TF.js），但研究使用率大幅下降。

JAX（Google）是一个新兴的替代方案，它结合了 NumPy 风格的 API + 函数式自动微分 + XLA 编译 + TPU 原生支持。很多 Google DeepMind 的项目在用 JAX。

对你来说：这门课使用 PyTorch，这是正确的选择。掌握好 PyTorch 的三层抽象（Tensor → Autograd → Module），你可以轻松迁移到任何其他框架。

为什么这一讲在 vibe coding 时代尤其重要？

在 AI 辅助编程越来越普遍的今天，框架层面的 API 调用是最容易被 AI 替代的——让 AI 帮你写一个训练循环、搭一个 ResNet，几秒钟就能完成。但是：

当训练出现 RuntimeError: CUDA out of memory 时，你需要理解 GPU 内存模型（Part 1-2 的知识）。当梯度变成 NaN 时，你需要理解 autograd 的计算图和梯度流（Part 6 的知识）。当模型在训练集上表现好但验证集上不行时，你需要知道 model.train() vs model.eval() 的区别（Part 8 的知识）。

这些"为什么"层面的理解，才是你的真正竞争力。
